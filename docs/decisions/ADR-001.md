# ADR-001: Choice of Serverless Express Architecture on Netlify

## Status
Approved

## Context
We need to host a backend API server for the LetsRead mobile application to support user registration, authentication, and catalog access. The key requirements are:
1. Minimizing server maintenance overhead (no OS updates, VM monitoring, etc.).
2. Cost efficiency (ability to scale down to zero when the application has low usage).
3. Fast and automated deployment pipelines linked to Git.
4. Ease of development—ensuring the team can write standard Express.js routes without learning proprietary serverless interface signatures.

## Decision
We chose to deploy the backend on **Netlify Functions** using an Express.js application wrapped with the `serverless-http` adapter.

- **Express.js Framework:** Provides familiar, mature middleware and routing capabilities.
- **Serverless-HTTP:** Automatically maps API Gateway events to Node.js `http.IncomingMessage` and `http.ServerResponse` objects, allowing us to export the entire Express application as a single Netlify Function handler (`functions/app.js`).
- **Netlify Hosting:** Handles builds, deployments, environment variable injection, and scaling automatically.

## Consequences

### Advantages:
- **Zero Server Management:** No need to configure Web servers (Nginx/Apache), setup firewalls, or manage load balancers.
- **Cost Efficiency:** Netlify functions are billed on execution time. The service scales to zero cost when idle.
- **Developer Familiarity:** Developers can write and test standard Express code locally, and use familiar routing, middleware, and request validation.
- **Unified Hosting:** Deployment is handled via a single configurations file (`netlify.toml`) linked to git repositories.

### Disadvantages / Risks:
- **Cold Starts:** When the function is idle, the first request will take longer as Netlify spins up a container and Mongoose reconnects to MongoDB Atlas.
- **Function Timeouts:** Netlify has a default execution timeout of 10 seconds. Long-running tasks or unindexed DB operations can trigger a 502 error if they exceed this limit.
- **Stateless Environment:** In-memory sessions, local web sockets, or local file system writes are not supported. State must be externalized to the MongoDB database or S3.
