# ADR-002: Choice of MongoDB and Mongoose ODM

## Status
Approved

## Context
The LetsRead application requires a data persistence layer that can handle:
1. User registration data and profile settings.
2. Dynamic catalog data (Book lists with descriptions, chapters, cover images, and genre tags).
3. User relationship histories (bookmarked books, reading histories).

The database must support rapid development, schema flexibility (e.g., adding metadata or new features like book review structures without running complex migrations), and run reliably on fully managed cloud infrastructure.

## Decision
We chose **MongoDB** (deployed via the fully managed MongoDB Atlas cloud service) along with **Mongoose ODM** for the Node.js application.

- **Document Model:** Aligning closely with JSON data structures makes transferring payloads from the server to the client trivial.
- **Mongoose ODM:** Provides structured schema modeling, built-in query validations, document middleware Hooks (such as pre-save hashing), and easy population of referenced tables (e.g., resolving book object IDs stored in the User's bookmark array).

## Consequences

### Advantages:
- **Flexible Schema:** Fields can be added dynamically (for instance, adding the `banner` property to the Book schema) without table lockouts or complex SQL migration scripts.
- **Document Nested Structures:** Storing arrays of IDs directly inside the user document (`bookmarks` and `recently`) simplifies queries and avoids many-to-many relationship join table overhead.
- **Managed Operations:** MongoDB Atlas provides automated backups, security patching, and scaling out of the box.

### Disadvantages / Risks:
- **Lack of Strict Relational Integrity:** Document references (`ref`) in Mongoose do not enforce foreign-key constraints. If a book is deleted from the `books` collection, any user bookmarks referencing that book's ID will point to a non-existent document. The application layer must handle cleanup or query validation manually.
- **Document Size Limits:** MongoDB documents are capped at 16MB. Storing arrays like `bookmarks` or `recently` directly in the user document works for small-to-medium datasets, but if a user bookmarks thousands of books, these arrays should be paginated or refactored into a separate collection.
